Architecture Overview

Status. Final draft for Data360 Global Challenge phase 2 (2026-05-29).
License. Code released under GPLv3; documentation under CC BY 4.0.

What this system is

Data360 News Agent (repo abrimos-data360-monitor) is an AI-powered news agency that detects newsworthy facts across the 12,000 development indicators of World Bank Data360 and delivers them — verified and with local perspective — to Latin American and Caribbean newsrooms.

The system treats Data360 as an authoritative source and turns the World Bank's catalogue into a discoverable, AI-ready feed for public-interest media. It monitors indicators, detects statistically and editorially significant changes, generates bilingual narratives (Noticia + Reportaje), and publishes them with Proof-Carrying Numbers (PCN) so every numeric claim resolves back to the originating Data360 observation in two clicks.

The framing follows Craig Hammer's lens on Data360:

"As AI capabilities continue to grow, data reliability, trustworthiness, and discoverability rise to the level of critical priorities for the World Bank Group." — Craig Hammer, World Bank, Data360 Global Challenge launch, 2026-02-16.

This architecture operationalises that principle: data reliability through the PCN trace; trustworthiness through interpretable, human-in-the-loop generation; discoverability through the agency feed that surfaces indicators the press has not yet covered.

What's new vs. what's reused — innovation map

The challenge's terms of reference distinguish substantial new development from minor modifications. This prototype combines battle-tested infrastructure with a substantial new analytical and verification layer built specifically for development indicators.

Block Status Why it matters
Agentic LLM framework (Llama, Mistral, Qwen, Claude, GPT-4 benchmarked) 🟢 Reused Open-source models prioritised over closed ones by design
Distribution / newsletter layer 🟢 Reused Day-one delivery to existing partner newsrooms
TeseoETL ingestion core 🟡 Roadmap (production elsewhere since 2019, OCDS pipelines) Demo uses lightweight fetch-data.js HEAD-probe path; TeseoETL slated as production ingestion engine
OpenSearch / RAG with document embeddings 🟡 Roadmap Demo reads from local CSV tiers; OpenSearch is the canonical observation store on the production roadmap
Data360 API connector (lib/data360-client.js, freshness probe, selective fetch) 🔴 New Country- and indicator-agnostic; built to abstract any Data360 endpoint
Five detection strategies for development indicators 🔴 New (analytical, not interface) Abrupt change, narrative–data divergence, invisible trend, cross-indicator anomaly, uncovered milestone
Fact-Checker Agent for numeric time-series claims 🔴 New Built for this prototype; rule-based gates + LLM reasoning with source citation obligation
Citizen narrative generator (Noticia + Reportaje) 🔴 New Bilingual, schema-validated, claim-bound
PCN (Proof-Carrying Numbers) integration 🟢 Adopted (World Bank standard) We do not invent the standard; we adopt it as the verification contract — every figure resolves to {indicator, country, date, value, endpoint}
Scoped chat per article + global analytical chat over Data360 🔴 New Discoverability surface, not interface decoration

The substantial new development is architectural and analytical, not cosmetic: five novel detection strategies, a dedicated cross-domain verification layer for numeric claims, and a citizen-narrative module that did not exist in our previous work.

Demo architecture (31 May 2026)

The submission demo runs in historical replay mode: CSV snapshots are fetched once, analysis runs offline, and the web app reads a static data/alerts.json. There is no NiFi orchestrator, no OpenSearch index and no continuous polling loop in the demo — these belong to the production roadmap below. Being explicit about this boundary is part of the integrity contract with the jury.

Two indicator-selection modes share the same pipeline:

  • Static watchlist — ~35 indicators in lib/watchlist.js (pulse, annual, forecast tiers on disk).
  • Dynamic discovery — datasets updated in the last N days via /data360/searchv2, expanded to indicator codes, stored as a fourth tier (dynamic). Default analysis uses NOTICIA_TIERS = ['dynamic'].

Data flow detail

  1. Freshness probe — conditional HEAD on bulk CSV blobs; writes data/changed-since.json. This is the dynamic, not static contact with the API: each run reconciles only what moved at the source.
  2. Selective fetch — downloads changed indicators; refreshes data/context/{COUNTRY}/{tier}.csv and data/snapshots/.
  3. News ingest (optional) — GDELT and/or Gemini-backed fetch to data/news/{COUNTRY}/{YYYY-MM}.jsonl for LLM context (passive context for the narrative agent, not a detection input).
  4. Detection — strategies 1 (abrupt change) and 4 (cross-indicator anomaly) over LAC CSV context in the demo; strategy 3 (invisible trend) is partially surfaced through the newsletter selection scoring; the remaining two patterns are scaffolded in lib/detect/ and activated on the production roadmap.
  5. Narrative generation with interpretability — one LLM call per Noticia; one per Reportaje when ≥2 Noticias share a dataset_id. Output is validated against a deterministic schema (Q1 claim trace, Q2 schema, Q4 bilingual length) before it can be emitted. No claim is generated without a verifiable {indicator, country, date, value, endpoint} tuple.
  6. Emit — merge into data/alerts.json; enrich with display metadata and canonical article URLs.

The five detection strategies — analytical innovation

The novelty of this prototype is not in the UI; it is in the editorial questions the system asks of an authoritative time-series feed. The five strategies map one-to-one to the patterns a development-data editor cares about:

# Strategy Demo status Editorial question
1 Abrupt change (lib/detect/z-score.js) ✅ Live "Did this indicator move beyond historical variance?"
2 Narrative–data divergence 🚧 Roadmap "Does the official narrative match what the indicator shows?"
3 Invisible trend 🟡 Partial (newsletter scoring, inverse-coverage signal) "Has a multi-country pattern emerged that no newsroom has covered?"
4 Cross-indicator anomaly (lib/detect/cross-indicator.js) ✅ Live "Do two indicators that usually move together suddenly diverge?"
5 Uncovered milestone 🚧 Roadmap "Has a country crossed a threshold that should have been news?"

Two of the five strategies are fully implemented (1 and 4); a third (invisible trend) is partially active through the newsletter selection logic, which surfaces multi-country patterns under-covered by the press via a composite score with an inverse-coverage signal; the remaining two are scaffolded and slated for the post-Challenge roadmap. This is the honest boundary — we report what the system can verify, not what it could rhetorically claim.

Fact-Checker Agent — verification by design

Interpretability is not a marketing add-on; it is a precondition for emission.

┌─────────────────────────────────────────────────────────┐
│ Fact-Checker Agent (lib/analysis/quality-validator.js, │
│                     lib/pcn-verify.js)                  │
│                                                         │
│  Rule-based gates (deterministic):                      │
│   • Q1  claim_id resolves to a Data360 observation      │
│   • Q2  output schema validates (bilingual fields, IDs) │
│   • Q4  word-count bounds per locale                    │
│                                                         │
│  LLM reasoning (with source citation obligation):       │
│   • numeric values match the cited indicator/date       │
│   • directionality of change matches the underlying CSV │
│   • when the indicator is absent for country/year, the  │
│     system declares the gap rather than imputing        │
│                                                         │
│  Default decision: REJECT. Emission requires PASS.      │
└─────────────────────────────────────────────────────────┘

The agent has veto power before publication. Honesty about absent data is a feature: when an indicator does not exist for a country-year, the system declares the gap rather than inventing a value. This aligns with the IFCN principles of transparency of methodology and openness about limits.

Human-in-the-loop

The system augments newsrooms; it does not replace editors. Every alert is published to the feed as a draft surface: partner editors read, edit and decide whether to use it in their own outlets. The UI is built around editing, not one-click publishing. The newsroom keeps the last call.

Components

Monitor web (data360-monitor.js)

Standalone Node.js HTTP server. Default port 8090 (D360_PORT). Serves:

  • Country picker and LAC front pages (newspaper layout)
  • SEO-friendly article URLs per Noticia/Reportaje
  • Indicators hub and per-IDNO detail pages
  • Global analytical chat with tool traces (the discoverability surface for Data360)
  • Scoped chat on article pages (generation context injected)
  • Newsletter edition preview and subscription modal
  • Static legal pages (/privacidad, /terminos, …)

Analysis pipeline (bin/generate-analysis.js, lib/analysis/)

  • Detection: lib/detect/z-score.js, lib/detect/cross-indicator.js (strategies 1 and 4 live; 2, 3 and 5 scaffolded)
  • Context: lib/analysis/context-builder.js — CSV tiers, indicator metadata, data dictionary, GDELT headlines, numbered claim candidates
  • Phase 1: lib/analysis/runner.js — Noticia per indicator/country
  • Phase 2: lib/analysis/reportaje-runner.js — Reportaje per dataset
  • Quality: lib/analysis/quality-validator.js, lib/pcn-verify.js
  • Output: data/alerts.json consumed by UI and chat tools

LLM layer — open-source first

The agentic framework benchmarks and orchestrates multiple models. Open-source models (Llama, Mistral, Qwen) are prioritised over closed ones (Claude, GPT-4) by design: closed models remain available as a fallback for benchmarking, but the production target is full self-hosting. This protects newsroom data sovereignty and keeps the system defensible under public scrutiny.

Data360 integration

  • REST (default for fetch/detection): lib/data360-client.js — search, metadata, CSV HEAD/GET, /data360/data
  • MCP (optional for chat): worldbank/data360-mcp on localhost; lib/mcp-client.js with REST fallback in lib/chat/tools.js

The connector is country- and indicator-agnostic. The demo covers five LAC countries because focus is a design choice, not an architectural restriction — Data360 covers 200+ countries and the same code path serves any of them.

Verification (PCN — Proof-Carrying Numbers)

Every numeric claim in narratives carries a claim_id resolvable to a Data360 observation. The pipeline drops alerts whose claims fail verification. The frontend renders {{claim:…}} tokens via @pcn-js/core patterns and local helpers, so a reader can travel from any number in the published text back to the originating indicator card and the Data360 source endpoint in two clicks. Source traceability is operational, not aspirational.

Newsletter (demo scope)

  • Fixture editions in data/newsletter/editions/
  • Subscription form → data/newsletter/subscribers.tsv (gitignored)
  • No mail transport in demo; production adds SMTP + List-Unsubscribe + unsubscribe links
  • Day-one distribution partners — Animal Político, Quinto Elemento Lab, Ojoconmipisto — are integrated as readers of the alert feed, not as a logo wall

Deployment

Two independent processes on the server:

Service Port Entry
Monitor 8090 node data360-monitor.js
Data360 MCP 8021 (localhost) uv run fastmcp … (optional, chat only)

Nginx terminates TLS and proxies only the monitor. MCP binds to 127.0.0.1; not exposed publicly. Staging may use HTTP basic auth — see infra/nginx/README.md.

The stack is self-hostable end to end, including the LLM layer. This is the unit-economic and sovereignty argument: a newsroom in the Global South can run the system on its own infrastructure without recurrent licensing costs and without sending editorial data to third-party APIs.

Environment: .env for AI_PROVIDER, CHAT_AI_PROVIDER, MCP_URL, NVIDIA_API_KEY, etc. (see .env.example).

Beyond the prototype

This architecture is the seed of a regional knowledge bank layer for Data360: a feed that turns the World Bank's catalogue into editable material for public-interest media across the Global South. The licence (GPLv3) guarantees that every derivative remains a public good. The connector is endpoint-agnostic by design, the verification layer is portable across numeric-claim domains, and the partner network is real. What ships on 31 May is the working core; what the architecture enables is a replicable model for AI-ready, authoritatively sourced civic journalism wherever Data360 reaches.
