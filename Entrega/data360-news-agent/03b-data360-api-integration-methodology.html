03b-Data360-API-Integration-Methodology

Data360 API Integration Methodology

Status. Final draft (2026-05-29).
Released under CC BY 4.0. Reference implementation released under GPLv3.

"As AI capabilities continue to grow, data reliability, trustworthiness, and discoverability rise to the level of critical priorities for the World Bank Group." — Craig Hammer, World Bank, Data 360 Global Challenge launch, 2026-02-16.

Purpose

This document specifies how the prototype treats the Data360 API as an authoritative source for an AI-ready newsroom pipeline. The integration is continuous and dynamic — scheduled polling against the live API, never static snapshots — so that every generated story is rebuildable from the API alone: indicator code, country code, date, value, endpoint. No intermediate cache, dump, or mirror substitutes for the source; the cached blobs described below are a freshness optimization, not a substitute. Data360 is the primary anchor of every published claim; supplementary sources (e.g. OCDS) only enter through a separate verification layer and never carry the analytical weight.

Operational reading: this is the deliverable that demonstrates Criterion 2 (Effective Use of World Bank Data360) and supports Criterion 5 (Feasibility & Technical Robustness). It aligns with Hammer's own framing — data quality, authoritative sourcing, AI-readiness, discoverability — and treats Data360 as the knowledge bank whose discoverability into LAC newsrooms is the system's reason to exist.

Endpoints used (Data360 API v3)

Endpoint Method Purpose
/data360/searchv2 POST Indicator discovery (also used by dynamic-watchlist mode, ordered by series_description/date_last_update desc)
/data360/metadata POST Full indicator metadata (OData syntax)
/data360/indicators GET List indicators per dataset. Returns the canonical idno used verbatim in URLs (never re-prefixed with the database id)
/data360/disaggregation GET Available disaggregations
/data360/data GET Observation values (LAC context files)
data360files.worldbank.org/.../data/{DB}/{IND}.csv GET / HEAD Bulk CSV download and freshness probe. The same prefix also serves {IND}_DATADICT.csv

URL convention

The bulk CSV URL is {BLOB_BASE}/{databaseId}/{idno}.csv, with idno used exactly as returned by listIndicators. The static watchlist happens to use idnos that start with the database id (e.g. WB_WDI_* under WB_WDI), but this is a property of the values the API returns, not a separate convention. Dynamic discovery follows the same rule. See csvUrl in lib/data360-client.js.

Freshness detection (Phase 1) — dynamic ingestion, not static snapshots

Data360 does not expose a bulk "updated since" feed. To keep ingestion dynamic — the system must react to upstream change, not consume a frozen export — the monitor detects upstream changes with conditional HEAD on each indicator's bulk CSV blob, on a scheduled cadence.

  1. Probe. HEAD the CSV URL with cached If-None-Match / If-Modified-Since.
  2. Interpret. 304 = unchanged. 200 = changed (or first probe).
  3. Report. Write data/changed-since.json with the list of changed idno values.
  4. Fetch selectively. Download CSV snapshots and refresh LAC context only for changed indicators.

The metadata field series_description.date_last_update is null for all probed watchlist indicators (2026-05-21). Do not use it for change detection. The reliable signal is the blob Last-Modified header.

See docs/data-fetcher-architecture.md for schemas, commands, and module map.

Detection strategies (demo)

  1. Abrupt changes. Z-score against historical trajectory per (indicator, country, disaggregation).
  2. Cross-indicator anomalies. Outliers from the regional pattern within the LAC scope.

MCP vs REST (chat runtime)

Use case Default path Optional MCP
Bulk CSV fetch, freshness probe REST (lib/data360-client.js) —
Detection, PCN verification Local CSV context —
Chat tools (search, get_data, compare) REST fallback always available lib/mcp-client.js when MCP_URL set

MCP runs on localhost only in production layout. Evaluate with node bin/evaluate-mcp.js.

Indicator links (D-035)

Public UI links use https://data360.worldbank.org/en/indicator/{IDNO} via lib/data360-urls.js and static/js/data360-urls.js. Dataset pages use /en/search?query={DATABASE_ID}.

Example: metadata query (OData-style)

POST /data360/metadata with filter on indicator idno — used when enriching indicator pages and context builder. See lib/data360-client.js → getMetadata().

Example: dynamic discovery

POST /data360/searchv2 with orderby: 'series_description/date_last_update desc', then GET /data360/indicators?datasetId=X for canonical idno values. Implemented in lib/dynamic-watchlist.js.

Reproducibility contract

Every alert and every citizen narrative produced by the system is independently rebuildable from the Data360 API alone. The verification trace attached to each claim records:

  • idno — the canonical indicator identifier as returned by listIndicators
  • country — ISO-3 code
  • date — observation timestamp
  • value — the observed figure
  • endpoint — the API path and parameters used to retrieve it
  • retrieved_at — UTC timestamp of the call

Any third party — a fact-checker at Poynter, an editor at a partner newsroom, a policymaker reviewing a brief — can replay the call and confirm or refute the figure. When the indicator does not exist for a given country-year, the system declares the gap explicitly; it does not interpolate, infer, or substitute. This is what AI-ready means operationally for this prototype: every model output is anchored to a retrievable record in the authoritative source.

How this advances Data360 as a knowledge bank

The integration is designed to widen Data360's reach into LAC newsroom workflows without re-mediating its authority. The connector is country-agnostic and dataset-agnostic; the MVP focuses on five LAC countries and a curated indicator family because we chose focus, not because the architecture restricts it. Each published story functions as a discoverability event for the underlying indicator — readers click from the story into the Data360 record in two steps — turning the knowledge bank into a feed that public-interest media can edit, contextualize, and distribute under their own editorial control. The reference implementation is GPLv3 so that every derivative remains a public good.
