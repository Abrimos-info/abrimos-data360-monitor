Security and Data Handling

Status. Final draft for Data360 Global Challenge phase 2 (2026-05-29).

Data360 News Agent is an AI-based news agency that detects newsworthy facts across the 12,000 indicators of Data360 and delivers them — verified and contextualised for LAC newsrooms. This document describes how the system handles data, user input, and third-party services across the pipeline that produces those alerts and citizen narratives. It is written for the Data360 Global Challenge jury.

The architecture is self-hosted by default: data sovereignty and cost predictability for public-interest newsrooms in the Global South are design constraints, not afterthoughts. The pipeline is built on battle-tested foundations — TeseoETL has been ingesting government data sources in production since 2019 — and shields every step from indicator ingestion to claim publication, so that Data360's authoritative sourcing remains intact as it flows toward the reader.

1. Data sources

Public indicator data (no PII)

All economic and social indicators come from World Bank Data360, treated as the authoritative primary source for every published claim. Ingestion is dynamic — not static snapshots: the connector polls the API on a schedule, so each generated story is rebuildable from the indicator code, country code, date and value alone. The demo does not ingest private datasets, credentials, or personally identifiable information from indicator sources.

New component (red block in the architecture overview): the Data360 connector — an extension of TeseoETL with parsers, normalisation and change detection tailored to development indicators. The rest of the ingestion stack is reused (green) and has been hardened over years of production use.

License compliance: each indicator carries a license field (typically CC BY 4.0). The pipeline preserves license references in alert metadata and verification traces, so that downstream readers can trace every value back to Data360 with data reliability, trustworthiness and discoverability intact.

Press headlines (public URLs)

Headlines are fetched from GDELT Document API v2 and/or Gemini-assisted search (see bin/fetch-news.js). Stored fields include title, URL, date, and source domain — all public web content. No paywalled full-text storage in the demo.

2. User-provided data

Newsletter and indicator alerts subscription

When a visitor submits the subscription modal:

Field stored Purpose
Email address Intended recipient for future newsletter/alerts
Subscription type newsletter_lac or indicator_alerts
Countries (optional) Filter scope for indicator alerts
Topics (optional) macro, fiscal, social, food_security, governance
Language preference es or en
Timestamp When the request was received
User-Agent, Referer Operational debugging only

Storage: append-only TSV at data/newsletter/subscribers.tsv. This file is gitignored and lives only on the server filesystem.

Demo scope (intentional minimum): a single real send is captured on the day of submission; no automated mailing, no double opt-in confirmation, no automated unsubscribe flow yet. The UI states this clearly; full policy text lives at /privacidad (loaded from config/copy/privacidad.{es,en}.json).

Production roadmap: self-hostable mail relay (no vendor lock-in), encrypted storage at rest, GDPR-style deletion on request, one-click unsubscribe in every message, and per-newsroom data residency for partners with local regulatory constraints.

Chat messages

Chat augments the journalist — it surfaces verified context from Data360, but every published claim still flows through the newsroom editor before it leaves the building. Questions typed in global or article-scoped chat are sent to the configured LLM provider for the current session. In demo:

  • Chat input is not linked to subscription email (no account system).
  • Conversation history for article chat is stored in browser sessionStorage only — not on the server after the SSE response completes.
  • Global chat history is held in page memory for the session; not persisted server-side.
  • No chat answer is auto-published; the chat is an editorial assistant, not an autopilot.

3. Third-party processing

LLM providers — open-source first

The pipeline is interpretable by design: every claim a model emits is anchored to a retrieved Data360 record, and no claim leaves the pipeline without a verifiable indicator, value, timestamp and API endpoint. Provider choice is configurable to avoid vendor lock-in, with a deliberate preference for self-hostable open-source models (Llama, Mistral, Qwen via LAIA / vLLM) over closed APIs (Claude, GPT-4) — the closed providers exist as benchmarking and fallback options, not as architectural dependencies.

Configured via environment variables — see the full secrets and configuration table in docs/environment-variables.md (repo) (AI_PROVIDER, CHAT_AI_PROVIDER, GEMINI_API_KEY, NVIDIA_API_KEY, etc.):

Provider Typical use Hosting Data sent
LAIA / vLLM (Llama, Mistral, Qwen) Chat and pipeline (preferred) Self-hostable User question + tool results
NVIDIA NIM Optional chat/pipeline Hosted, swap-in Same as above
Claude (Agent SDK) Pipeline narratives, benchmarking Hosted Omnibus analysis context (public data + headlines)
OpenRouter Fallback Hosted Same as above

Tool results sent to any LLM contain public indicator values and alert summaries — never subscriber emails. Operators reviewing the system for a specific jurisdiction (EU, Brazil, Singapore) can run the entire stack against self-hosted open-source models with no functional loss, which is the regulatory escape hatch built into the architecture.

Data360 MCP server

Optional Python MCP server (worldbank/data360-mcp) runs on localhost only (127.0.0.1:8021). The Node monitor calls it for chat tools; it is not exposed through nginx or the public internet.

Gemini (news fetch)

When npm run fetch:news uses Gemini, article titles/URLs may be sent to Google's API for grounding or summarization. API keys belong in .env (gitignored).

4. Cookies, tracking, and analytics

The demo does not use:

  • Third-party analytics (Google Analytics, Meta Pixel, etc.)
  • Advertising trackers
  • Cross-site tracking cookies

Language preference and onboarding dismissal use first-party browser storage (localStorage / query params) only. No cookie banner is required for the demo scope; the privacy page at /privacidad documents this.

5. Claim verification (PCN)

Every numeric value in generated narratives maps to a verifiable claim token with a claim_id. Information integrity is operationalised, not invoked: the system refuses to publish any claim without a verifiable Data360 anchor.

Claim ID formats

Source Length Algorithm
Data360 MCP (mcp_get_data) 8 hex chars Server-side opaque ID (passed through)
Local CSV (pipeline) 16 hex chars SHA-256 over sorted JSON {indicator, country, time_period, unit_measure}, truncated

Verification pipeline

lib/pcn-verify.js checks each claim against data/context/{COUNTRY}/{tier}.csv for tiers in CONTEXT_TIERS (annual, forecast, dynamic). A 0.1% numeric tolerance handles rounding. MCP-sourced 8-char IDs skip ID consistency check and compare values directly. Alerts with any unverified claim are dropped before writing data/alerts.json.

Frontend

@pcn-js/core and @pcn-js/data360 (Apache-2.0) support claim mark rendering. Every published claim is independently verifiable in two clicks: indicator card → Data360 source URL in verification_trace. This is the user-facing surface of the Shielding the Pipeline principle: the reader, not just the editor, can audit any number on the page.

6. Hosting and transport

  • Infrastructure: Abrimos-owned servers in the demo; the deployment recipe is fully self-hostable by any partner newsroom, with no proprietary dependencies along the critical path. Data sovereignty for newsrooms in developing countries is the default, not an upgrade tier.
  • TLS: HTTPS in production/staging via nginx.
  • Staging gate: optional HTTP basic auth on the monitor only (infra/nginx/.htpasswd — demo credentials documented as intentionally weak).
  • Secrets: API keys and provider tokens in .env, never committed (see .gitignore). Full variable list: docs/environment-variables.md (repo).
  • Static assets: served from /static/* with cache headers; no user data in static files.

MCP and subscriber TSV are not reachable from the public internet in the recommended deployment layout.

7. Repository and license

  • Source: abrimos-info/abrimos-data360-monitor under GPLv3 — a deliberate choice, not a default. Strong copyleft ensures every derivative remains a public good and the core verification layer stays freely accessible to public-interest media in the Global South.
  • Generated alerts and analysis artifacts in data/ may be committed for demo replay; subscriber TSV is excluded.
  • Contact for data requests: contacto@abrimos.info (referenced on /privacidad).

9. Beyond the prototype — what shielding the pipeline buys

The protocols above are not a compliance exercise. They are the substrate that makes the rest of the proposal credible: a self-hostable, open-source, AI-ready agent that turns Data360 — the World Bank's authoritative knowledge bank — into a discoverable feed for LAC newsrooms, without forcing those newsrooms to surrender data sovereignty or editorial independence in exchange for the tool.

That substrate is what allows partner newsrooms to adopt the agent without renegotiating their own governance, and what allows the same architecture to replicate to other regions and other indicator families with no change to the security model. Reliability, trustworthiness and discoverability — the criteria the World Bank has named as critical priorities — are protected end-to-end: from API call to claim mark on the published page.
